1.2 State Transitions
Enrollment Workflow

Student POSTs to /api/courses/{slug}/enroll/ → status='pending'

Tutor GETs pending via /api/enrollments/?course=slug&status=pending/ → PUT /api/enrollments/{id}/ with status='active' or 'rejected'

Frontend moves student to course home on 'active'.

Assignment Submission → Grading

Student GETs assignments list for active course modules.

Student POSTs to /api/assignments/{id}/submit/ → creates Submission.

Tutor retrieves via /api/submissions/?assignment=id → PUT /api/submissions/{id}/ with feedback, score, and graded=true.

Frontend shows feedback immediately.

Badge Awarding

On enrollment completion or assignment grading, a Django signal checks Badge.criteria and issues badges to User.badges (ManyToMany if you like).

5. Offline Sync Logic (IndexedDB)
Cache Data: On every GET, write to IndexedDB (courses, modules, assignments).

Queue Writes: On POST/PUT when offline, enqueue operations in a “pendingOps” store.

Sync Daemon: On navigator.onLine event, drain pendingOps in sequence, retry on failure.

Conflict Handling: Use updated_at timestamps; if server version is newer, alert the user.

const DB_NAME = 'learn-smart-db';
const STORE_CACHE = 'cache';

async function initDB() {
return openDB(DB_NAME, 1, {
upgrade(db) {

db.createObjectStore(STORE_OPS, { autoIncrement: true });
}
